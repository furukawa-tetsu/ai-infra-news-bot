## 1. Slack投稿 → Obsidian保存フロー検討

### やりたかったこと

Slack投稿から、記事をObsidianへ詳細解説付きで保存したい。

### 検討した案

#### 案A: Slack Button方式

Slack投稿にBlock Kit Buttonを埋め込み、クリック時にCloudflare Workersへ通知する方式。

ただし、この方式は以下が必要。

- Slack App作成
- Interactive Components設定
- Request URL設定
- 管理者承認

そのため、今回は見送り。

#### 案B: RSS本文に保存リンクを埋め込む方式

Slack RSS Appの投稿本文に以下のような保存リンクを埋め込む。

`https://ai-infra-news-bot.ai-infra-news.workers.dev/save?id=<article_id>`

このリンクをクリックすると、Workers側で記事を取得し、Obsidian用Markdownを生成する。

今回はこちらを採用。

## 2. `/save?id=...` エンドポイント実装

### 実装内容

Cloudflare Workersに以下のエンドポイントを追加。

`/save?id=<article_id>`

### 処理内容

1. query parameterから `id` を取得
2. D1の `articles` テーブルから記事を取得
3. Obsidian用Markdownを生成
4. `text/markdown` として返却

### 追加した主な関数

- `generateObsidianMarkdown(article)`
- `escapeYaml(value)`

### Markdownに含める内容

- title
- source
- url
- published
- category
- relevance
- summary
- implication
- tags

---

## 3. `feed.xml` に保存リンクを追加

### 修正内容

RSSの `<description>` 内に、Obsidian保存用リンクを追加。

例:

`Obsidian保存用Markdown: https://ai-infra-news-bot.ai-infra-news.workers.dev/save?id=<article_id>`

### SQL修正

`generateFeedXml()` 内のSELECT対象に `id` を追加。

```sql
SELECT id, title, url, source, published_at, summary, category,
       relevance_score, implication, created_at
FROM articles
```

## 6. R2保存機能追加

### 実装内容

`/save?id=...` 実行時に、MarkdownをCloudflare R2へ保存するように拡張。

### 処理フロー

1. D1から記事を取得
2. 未保存ならOpenAIで詳細解説Markdownを生成
3. R2へMarkdownを保存
4. D1に保存パスと保存時刻を記録
5. Markdownをレスポンスとして返却

### R2保存パス

```text
obsidian/YYYY-MM-DD/<slug>-<article_id>.md
```

## 7. D1カラム追加

R2保存状態を管理するため、`articles` テーブルへカラムを追加。

```sql
ALTER TABLE articles ADD COLUMN saved_markdown_path TEXT;
ALTER TABLE articles ADD COLUMN saved_at TEXT;
```

### migration適用

ローカル:

```bash
npx wrangler d1 migrations apply ai_infra_news --local
```

本番:

```bash
npx wrangler d1 migrations apply ai_infra_news --remote
```

---

## 8. R2保存済み記事の再利用

### 実装内容

同じ `/save?id=...` が再度クリックされた場合は、OpenAIを再実行しない。

### 処理

1. D1の `saved_markdown_path` を確認
2. すでに保存済みならR2からMarkdownを取得
3. Markdownを返却

これにより、同じ記事で不要なOpenAI APIコールを防止。

## 10. rclone導入とR2接続

### rcloneインストール

```bash
curl https://rclone.org/install.sh | sudo bash
```

### R2用remote作成

```bash
rclone config
```

### 発生したエラー

当初、以下のようにAWS S3へ接続しようとして失敗。

```text
https://s3.auto.amazonaws.com/ai-infra-news-md/obsidian
```

### 原因

rcloneのR2設定で `endpoint` が未設定だった。

### 修正

Cloudflare R2用endpointを追加。

```text
endpoint = https://<ACCOUNT_ID>.r2.cloudflarestorage.com
```

### 接続確認

```bash
rclone lsd r2:
rclone lsd r2:ai-infra-news-md
rclone ls r2:ai-infra-news-md/obsidian
```

## 12. `sync-obsidian.sh` 作成

### 目的

同期処理をワンコマンド化する。

### 作成ファイル

`scripts/sync-obsidian.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail

DEST="/mnt/c/Users/古川哲/Documents/Note/Articles/AI-Infra-news"

mkdir -p "$DEST"

mkdir -p logs

echo "==== $(date) ====" >> logs/sync.log

rclone sync \
  r2:ai-infra-news-md/obsidian \
  "$DEST" \
  --include "*.md" \
  --create-empty-src-dirs \
  >> logs/sync.log 2>&1
```

### 実行権限付与

```bash
chmod +x scripts/sync-obsidian.sh
```

### 実行

```bash
./scripts/sync-obsidian.sh
```

## 14. Windows Task Schedulerによる定期同期設計

### 目的

R2からObsidian Vaultへの同期を定期実行する。

### ローカル同期時刻

```text
07:10
12:10
18:10
```

Cloudflare Workers側のCron実行時刻より少し後に同期する設計。

### WSL実行コマンド

PowerShell / Windows Task Schedulerから以下を実行。

```powershell
wsl.exe -d Ubuntu -- bash -lc "cd ~/ai-infra-news-bot && ./scripts/sync-obsidian.sh"
```

### WSLディストリ名確認

```powershell
wsl -l -v
```

### Task Scheduler設定方針

3つのタスクを作成するのが確実。

| タスク名 | 実行時刻 |
|---|---:|
| Sync AI Infra News 07 | 07:10 |
| Sync AI Infra News 12 | 12:10 |
| Sync AI Infra News 18 | 18:10 |

### 操作設定

プログラム:

```text
wsl.exe
```

引数:

```text
-d Ubuntu -- bash -lc "cd ~/ai-infra-news-bot && ./scripts/sync-obsidian.sh"
```

## 所感

今回の拡張により、単なるRSS Botではなく、以下に近い構成になった。

```text
AI Infrastructure Intelligence System
```

特に重要だった設計判断:

- Slack App承認を避けるため、RSS本文に保存リンクを埋め込む

現在の構成は、Cloudflare Workers / OpenAI / D1 / R2 / Slack RSS / Obsidian を組み合わせた、軽量なAI知識パイプラインとして成立している。
